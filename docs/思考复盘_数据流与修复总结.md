# 思考复盘：数据流与修复总结

## 一、本阶段完成事项

### 1. 五类用户侧问题修复（登录与数据加载）

| 问题 | 根因 | 修复方式 |
|------|------|----------|
| 输错密码没报错 | 错误对象用 `error?.response?.code`，PB 用 `error.status`；未覆盖所有错误类型 | 统一用 `error.status`/`error.response?.code` 判断，默认提示「用户名或密码错误」 |
| 记住登录没记住 | 只存了 username，未控制 token 持久化 | 用 `rememberMe` 写 localStorage；未勾选时用 sessionStorage 做会话检测，新会话清 authStore；logout 时清标记 |
| 登录后显示网络错误要手动刷新 | 登录成功后未清 query 缓存，旧失败结果残留 | 登录成功后在 catch 外调用 `queryClient.clear()` |
| 横屏白屏 | `innerWidth > 768` 在横屏手机也为 true，误用 PC 侧栏布局 | 用「宽度 + 触屏 + 高度」综合判断 isPC；横屏手机 (h<500) 不按 PC 处理；Home 容器改为 100dvh |
| 数据流处处加载失败 | 未登录时 query 仍发请求导致 401 | 所有 useProjects/useTasks/useUsers 等加 `enabled: pb.authStore.isValid` |

## 二、「三个地方数据不一样」的根因分析

### 2.1 三个入口各自的数据来源

| 入口 | 数据来源 | 项目列表含义 | 任务数据 |
|------|----------|--------------|----------|
| **工作进展 (Tasks.tsx)** | `useProjects()` + `useTasks()` | `projectsRaw.filter(p => p.status !== 'archived')` → **仅展示 5 个活跃项目** | 全量任务（经理）或按 assignees 过滤（员工） |
| **管理控制台概览 (AdminDashboard)** | 本地 `loadData()`：`getFullList` 用户/项目/任务，**无 status 过滤** | `totalProjects = projects.length` → **10（含归档）**；「项目进度」用 `projects.slice(0, 5)` → **只取前 5 条** | 全量 82 条任务 |
| **经理工作台 (ManagerDashboard)** | `useProjects()` + `useTasks()` | `totalProjects = projects.filter(p => p.status === 'active').length` → **5** | 同 api 的 useTasks |

### 2.2 不一致的具体表现

1. **项目总数 vs 项目进度条数**
   - 概览卡「项目总数」= **全部项目数**（10，含 5 个 archived）。
   - 「项目进度」只渲染 **前 5 个项目**（`slice(0, 5)`），且未先过滤 archived。
   - 结果：用户看到「总数 9/10」和「下面只有 5 条进度」，语义不统一。

2. **数据源不统一**
   - 工作进展 / 经理工作台：**统一用 api 的 useProjects/useTasks**（同一套 queryKey，同一份缓存）。
   - 管理控制台：**单独 loadData()**，不走 react-query，数据与上面两处**不同源、不同步**，易出现「概览是旧数据、其他页已更新」或数量口径不一致。

3. **时间轴「1个任务 / 10逾期」之类错误**
   - 若某处按「项目维度的任务数」与「逾期数」分别统计且来源不一致（例如一个用 `tasks.length`，一个用全局 overdue 总数），就会出现「1 个任务却 10 逾期」的悖论。需要核对时间轴/概览里「逾期」的统计口径（按项目 vs 全局、是否过滤 status）。

### 2.3 根因归纳

- **统计口径不统一**：有的用「全部项目」，有的用「仅 active」，有的用「前 5 条」。
- **数据源分裂**：AdminDashboard 用本地 getFullList，其它用 useProjects/useTasks，没有单一数据源。
- **展示与总数未对齐**：「项目进度」只展示前 5 条，却未说明「项目总数」是否含归档、是否与这 5 条一致。

## 三、建议的修复方向（后续可做）

### 3.1 统一「项目总数」与「项目进度」口径（AdminDashboard）

- **方案 A**：概览「项目总数」改为**仅统计活跃项目**（与经理工作台一致），并注明「活跃」：  
  `totalActiveProjects = projects.filter(p => p.status !== 'archived').length`  
  「项目进度」改为：先 `filter(p => p.status !== 'archived')` 再 `slice(0, 5)`，这样条数与「总数」语义一致（至少前 5 条都来自同一集合）。
- **方案 B**：保留「项目总数」= 全部（含归档），但在 UI 上区分「全部 10 / 活跃 5」，且「项目进度」下写「活跃项目进度（前 5）」，避免误解。

### 3.2 管理控制台尽量复用 api 层

- 将 AdminDashboard 的 `loadData()` 改为使用 `useProjects()`、`useTasks()`、`useUsers()`（或为 admin 单独开一组 queryKey 但同一套 queryFn），这样：
  - 与工作进展、经理工作台**同一数据源、同一缓存**；
  - 数量与过滤逻辑可集中写在 api 或少量 useMemo 里，避免「三处三种算法规矩」。

### 3.3 时间轴/概览的「逾期」统计

- 明确「逾期」定义：**按任务维度**（`status === 'overdue'` 或 `deadline < today`）按项目聚合。
- 时间轴每个项目卡片：**任务数**与**该项目的逾期数**必须来自同一份任务列表（同一 `tasks.filter(t => t.project === id)`），避免混用全局逾期数导致「0 任务却 10 逾期」。

### 3.4 数据库与前端口径对齐（可选）

- 当前 PB 里 `projects` 的 `total_tasks` / `completed_tasks` / `progress` 可能与实际 `tasks` 表不一致（例如手工改过任务未回写项目）。
- 若希望「项目进度」与「任务状态」完全一致，可考虑：  
  - 要么**以 tasks 实时聚合为准**（前端按 tasks 算 progress），  
  - 要么在服务端/钩子里**按任务表重算并回写** projects 的 progress/total_tasks/completed_tasks，再供前端展示。

## 四、复盘与原则总结

1. **单一数据源**：同一类数据（项目/任务）尽量只从一处拿（同一 api 层、同一 queryKey），避免「概览一套、列表另一套」。
2. **统计口径一致**：所有「总数」「进度」要明确是「全部」还是「仅活跃」；若只展示前 N 条，需在文案或逻辑上与总数口径一致。
3. **权限与 enabled**：依赖登录态的数据请求统一加 `enabled: pb.authStore.isValid`，避免未登录 401 和残留错误态。
4. **登录后清缓存**：登录成功执行 `queryClient.clear()`，避免旧身份/失败请求的缓存影响新身份。
5. **测试与构建**：改动后跑 `tsc --noEmit` 与 `npm run test`，再 `npm run build`；测试里对只读的 pb.authStore 用 `vi.hoisted()` 提供可写的 mock，避免只读属性报错。

当前代码已按上述思路完成登录与数据加载类修复，并通过构建与测试。数据口径统一与 AdminDashboard 改用 useProjects/useTasks 可作为下一阶段优化项，按需实施。
