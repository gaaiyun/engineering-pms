## 二、Kaneo 深度分析

**仓库：** https://github.com/usekaneo/kaneo
**已 Clone 到：** `g:\kaneo`

### 2.1 架构亮点

**Monorepo + Turborepo 结构**

```
kaneo/
├── apps/
│   ├── web/          # React 19 + Vite 7 + TanStack Router
│   ├── api/          # Hono + Drizzle ORM + PostgreSQL
│   ├── site/         # 营销网站
│   └── docs/         # 文档
├── packages/
│   ├── libs/         # 共享 Hono RPC 客户端
│   ├── email/        # React Email 模板
│   └── mcp/          # AI MCP 集成
└── i18n/             # 国际化
```

**端到端类型安全：** API 使用 Hono 框架，前端通过 `hc<AppType>` RPC 客户端直接获取类型推导，无需手写接口定义。这比我们手动定义 TypeScript 接口再调用 PocketBase 更安全。

### 2.2 通知系统设计

Kaneo 的通知采用 **事件驱动架构**：

```
领域操作 → publishEvent('task.status_changed') → EventEmitter 监听 → 创建通知记录 → 异步投递（邮件/其他渠道）
```

- **发布-订阅模式：** 使用 Node.js `EventEmitter` 解耦业务逻辑和通知逻辑
- **通知偏好：** 支持用户级别的通知渠道偏好设置（邮件 vs 站内）
- **排除自我通知：** 在事件处理器中过滤 `actor ≠ assignee`

**对我们的启发：** 我们目前的通知创建散落在各个 mutation 中（`notifyProjectMembers`、`notifyTaskAssignees` 等），每个调用点都要记得正确传参。应该考虑引入事件总线模式，将通知创建从业务逻辑中解耦。

### 2.3 进度计算

Kaneo 在 **API 端** 计算项目进度：

```typescript
// 服务端一次性计算，返回给前端
const completedTasks = project.tasks.filter(
  task => task.status === "done" || task.status === "archived"
).length;
const completionPercentage = totalTasks > 0
  ? Math.round((completedTasks / totalTasks) * 100) : 0;
```

**对我们的启发：** 我们之前的 bug 正是因为进度计算不统一——有的页面读 PocketBase 字段，有的页面客户端计算。Kaneo 的做法是在 API 响应中直接附带统计数据，前端只管显示。

### 2.4 看板列与任务状态

Kaneo 的看板列是 **数据库一等公民**：每个项目有自己的列（columns），任务的 `status` 就是列的 `slug`。这意味着：
- 状态是动态可配置的，不是写死的枚举
- 拖拽改变列就是改变状态
- 有"虚拟状态"概念（`planned`、`archived`）不对应任何可见列

**对我们的启发：** 我们的任务状态目前是固定的 `pending | in_progress | blocked | completed | overdue`。如果未来需要支持自定义工作流，可以参考 Kaneo 的 column-as-status 模式。

### 2.5 其他值得参考的设计

- **TanStack Router 文件路由：** 路由从文件结构自动生成，减少手动配置
- **搜索参数驱动弹窗：** URL query 中带 `taskId` 即可打开任务详情抽屉，支持浏览器前进/后退
- **Query 默认配置：** 关闭 `refetchOnWindowFocus` 和 `refetchOnMount`，减少不必要的请求；对网络错误不重试
- **30s 轮询替代 WebSocket：** 看板数据每 30s 自动刷新，实现简单且够用

## 三、Worklenz 深度分析

**仓库：** https://github.com/Worklenz/worklenz
**已 Clone 到：** `g:\worklenz`

### 3.1 架构亮点

```
worklenz/
├── worklenz-frontend/     # React 19 + Redux Toolkit + Ant Design
│   ├── src/features/      # 按功能模块组织
│   ├── src/pages/         # 路由页面
│   ├── src/api/           # Axios API 客户端
│   └── src/socket/        # Socket.IO 客户端
└── worklenz-backend/      # Express + TypeScript + PostgreSQL
    ├── src/controllers/   # 业务控制器
    ├── src/socket.io/     # 实时命令处理
    └── database/          # SQL 迁移和函数
```

### 3.2 通知系统（最值得参考）

Worklenz 的通知系统是我们见过的最完善的：

**三层通知架构：**
1. **站内通知（实时）：** Socket.IO 推送，用户立即看到
2. **邮件通知：** Pug 模板渲染 HTML 邮件，支持任务分配/完成/每日摘要
3. **用户偏好：** `email_notifications_enabled` / `popup_notifications_enabled` 独立控制

**通知创建流程：**
```
任务操作 → NotificationsService
  ├── 写入 DB (create_notification SQL 函数)
  ├── Socket.IO 推送到在线用户 (IO.emit NOTIFICATIONS_UPDATE)
  └── 异步发邮件 (sendAssignmentUpdate / sendTaskDone)
```

**对我们的启发：**
- 我们完全缺少 **用户通知偏好** 功能，应该让用户选择接收哪些类型的通知
- 通知创建放在 **SQL 函数** 中是个好主意，可以确保数据一致性
- **每日摘要邮件** 是很实用的功能，适合经理角色

### 3.3 进度计算（最完善）

Worklenz 提供 **三种进度模式**，用户可以切换：

| 模式 | 说明 |
|------|------|
| `use_manual_progress` | 手动设置百分比 |
| `use_weighted_progress` | 任务带权重，加权计算 |
| `use_time_progress` | 基于时间投入计算 |

**进度计算放在 PostgreSQL 函数中：** `get_task_complete_ratio` 处理了多级子任务嵌套、除零保护、归档排除等边界情况。经过多次迁移修复：
- `fix-multilevel-subtask-progress-calculation.sql`
- `fix-division-by-zero-in-task-ratio.sql`
- `update-time-based-progress.sql`

**对我们的启发：** 我们的进度计算逻辑非常简单（完成任务数/总任务数），缺少权重和子任务概念。如果未来任务层级变深，需要参考 Worklenz 的递归计算方式。

### 3.4 Socket.IO 实时协作

Worklenz 大量使用 Socket.IO 实现实时功能：

- **看板拖拽同步：** 一人拖拽，其他人实时看到
- **甘特图拖拽：** 调整时间线立即同步
- **任务字段编辑：** 状态、优先级、负责人变更实时推送
- **在线状态：** 用户登录时注册 `socket_id`

**命令模式：** 每个 Socket 事件对应一个独立的命令文件：
```
socket.io/commands/
├── on-task-status-change.ts
├── on-task-assignees-change.ts
├── on_gannt_drag_change.ts
├── on-update-task-progress.ts
└── ...
```

**对我们的启发：** PocketBase 的 realtime 订阅已经覆盖了基本的数据同步，但缺少 **细粒度的操作推送**。如果未来需要更精细的协作体验，可以考虑在 PocketBase 之上增加 Socket.IO 层。

### 3.5 资源管理

Worklenz 有完整的资源分配视图：
- 按项目查看资源分布
- 按人员查看工作负载
- 日/周/月视图切换
- SQL 函数 `get_project_wise_resources` / `get_team_wise_resources` 提供聚合数据

**对我们的启发：** 我们目前缺少 **人员工作负载视图**。经理需要看到每个员工手上有多少任务、是否超负荷。

## 四、Plane 在线分析

**仓库：** https://github.com/makeplane/plane (47,000+ Stars)

### 4.1 核心特点
- **最接近 Jira 的开源替代品**，功能最全
- Sprint/Cycle 管理、燃尽图
- 内置分析仪表板
- 自定义属性系统
- **Monorepo + Django 后端 + React 前端**

### 4.2 对我们的参考价值
- **自定义字段系统：** 允许用户给任务添加自定义属性，适应不同项目需求
- **模块化视图：** 列表/看板/日历/甘特图多种视图切换
- **筛选器系统：** 复杂的多维度过滤，保存为自定义视图
- **文档系统（Pages）：** 内置 Wiki 功能

**局限：** 技术栈差异大（Python/Django 后端），架构过重（13 个 Docker 服务），不适合直接参考代码，但产品设计理念值得借鉴。

## 七、对我们后续改进的建议

### 高优先级（短期可落地）

1. **引入通知偏好设置**
   - 参考 Worklenz：让用户选择开关不同类型的通知
   - 在 Profile/Settings 页面增加通知偏好面板
   - PocketBase 中增加 `notification_preferences` 集合

2. **进度计算进一步优化**
   - 参考 Kaneo：在 API 响应中附带统计信息而不是前端各自计算
   - 可以在 PocketBase 的 `projects` 集合中增加 hook 自动更新进度字段

3. **通知事件解耦**
   - 参考 Kaneo 的事件总线模式
   - 将通知创建从各个 mutation 中抽离到统一的事件处理器
   - 减少遗漏和不一致

4. **任务优先级完善**
   - 参考 Worklenz：为优先级添加颜色系统和排序权重
   - 高优先级任务在各视图中突出显示

### 中优先级（中期规划）

5. **自定义任务状态**
   - 参考 Kaneo 的 column-as-status：让经理自定义项目的工作流列
   - 保留系统默认状态，允许扩展

6. **人员工作负载视图**
   - 参考 Worklenz 的资源管理：经理可查看每人任务分布
   - 帮助避免任务分配不均

7. **每日/周工作摘要**
   - 参考 Worklenz 的 `sendDailyDigest`
   - 定时汇总未完成任务、即将到期任务、新通知
   - 可先用站内通知实现，后续扩展到邮件

8. **子任务支持**
   - 参考 Worklenz 的多级子任务
   - 允许将大任务拆分为子步骤，独立追踪进度

### 低优先级（长期愿景）

9. **URL 驱动的弹窗状态**
   - 参考 Kaneo 的 search param 模式：`?taskId=xxx` 打开详情
   - 支持分享链接直接打开特定任务

10. **端到端类型安全**
    - 参考 Kaneo 的 Hono RPC 客户端
    - 如果 PocketBase 支持，可以自动生成 TypeScript 类型

11. **外部渠道通知**
    - 参考 Planka 的多渠道支持
    - 集成企业微信/钉钉推送
