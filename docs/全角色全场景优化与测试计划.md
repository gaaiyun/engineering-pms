## 二、团队分工与职责

### 2.1 开发团队（Dev Team）

- **API 与数据流**：确保所有 mutation 后正确 `invalidateQueries`，关键集合已接入 PocketBase Realtime（SSE），通知/审计与业务操作一致。
- **单元/组件/集成测试**：维护并扩展 Vitest 用例，保证回归安全。
- **E2E 与自动化**：按角色与场景编写 Playwright 用例（登录、项目 CRUD、任务、看板、时间轴、通知、审批）。
- **性能与安全**：请求防抖、敏感数据不落前端、AI Key 仅服务端等。
- **Bug 修复与重构**：按产品验收结果修缺陷，在不改变产品语义前提下做必要重构。

### 2.2 产品经理团队（PM Team）

- **全角色场景验收**：以经理与员工两种角色走完核心路径，确认权限、可见性、操作边界。
- **需求对照**：对照 PRD/老板需求逐条检查（归档、分类、批量任务、审批界面、消息即时性、卡点简化等）。
- **交互与体验**：反馈是否“第一时间刷新”、导航是否一致、加载/空态/错误提示是否清晰。
- **审美与“非 AI 化”**：检查是否出现通用模板感、是否保持品牌化（如 Plus Jakarta / Slate 体系）、概览是否可点击穿透、信息层级是否清晰。

## 三、全角色矩阵

| 角色         | 可见范围           | 典型能力 |
|--------------|--------------------|----------|
| 经理/管理员  | 全部项目、管理后台 | 创建/编辑/删除项目，增删改成员，创建/编辑/删除任务，分配人员，归档，审批复核，批量任务编辑 |
| 普通员工     | 仅参与的项目       | 查看项目与任务、上报卡点；**不可**编辑任务/分配下一人/编辑项目 |

**必验权限边界**：

- 员工账号不出现「管理」Tab、不进入管理后台。
- 员工在项目详情/任务详情中无「编辑任务」「分配人员」「删除任务」等经理专属按钮。
- 经理从管理后台返回首页后，仍保留「管理」入口（无丢失身份问题）。

## 四、全场景测试清单（产品 + 开发共用）

### 4.1 认证与首页

- [ ] **登录**：正确账号密码可登录；错误密码有明确提示。
- [ ] **注册**：流程可用，无“密码输两遍”等文案/逻辑错误。
- [ ] **首页 - 经理**：底部 Tab 含「工作进展」「管理」「我的」；概览数据存在且可点击穿透（若有设计）。
- [ ] **首页 - 员工**：仅「工作进展」「我的」；展示参与项目的任务与最近未读消息摘要。
- [ ] **未读角标**：有未读通知时角标正确，进入通知页或标记已读后更新。

### 4.2 项目列表与分类

- [ ] **分类 Tab**：「全部」「进行中」「卡顿/阻塞」「已归档」展示与筛选正确。
- [ ] **排序**：支持按时间等维度排序。
- [ ] **归档**：经理可归档/取消归档；归档项目在概况统计中排除，在列表中仍可选「已归档」查看。
- [ ] **成员**：经理可进入项目成员管理，增删改成员；员工仅能看到自己所在项目。

### 4.3 任务与看板

- [ ] **创建任务**：经理可创建，关联项目与执行人；创建后列表/看板即时刷新。
- [ ] **批量任务编辑**：经理在项目内可打开批量编辑弹窗，三列（任务、执行人、时间）可编辑并一次性保存；保存后看板/时间轴即时更新。
- [ ] **看板**：拖拽更新状态后，数据与 SSE 一致，无需要手动刷新才看到的情况。
- [ ] **任务详情**：经理可编辑、分配、删除任务；员工仅可查看与上报卡点。
- [ ] **卡点**：员工只需填写原因；取消卡点时可选择变为「已完成」或「进行中」。

### 4.4 时间轴

- [ ] **移动端布局**：日期与任务对齐、不拥挤。
- [ ] **横屏**：横屏时时间轴可用、布局合理。
- [ ] **数据**：与项目任务数据一致，实时更新。

### 4.5 通知与审批（审计）

- [ ] **通知**：项目内任意变动（任务状态、人员、时间、项目编辑等）对项目内全员产生通知；经理变更计划后通知即时触发，前端列表实时刷新（SSE/轮询）。
- [ ] **审批界面**：作为变更审计与复核中心，展示项目名、操作人、任务、时间、改动内容；提供「已阅读」「通过」；支持筛选、分类、搜索；UI 为列表/卡片，数据即时同步、数字自动刷新。

### 4.6 数据流与 API 自检（开发侧）

- [ ] **Mutation 后 invalidate**：创建/更新/删除 项目、任务、成员、归档、卡点、交接 等操作后，对应 `queryKeys` 已 invalidate，界面不再依赖手动刷新。
- [ ] **Realtime 订阅**：`subscribeToChanges` 覆盖 `tasks/projects/notifications/audit_logs/...`，且登录后正确建立、登出或卸载时 `unsubscribeAll`。
- [ ] **审计与通知一致性**：关键操作（如任务完成、任务删除、项目删除、成员变更、卡点上报）同时写审计日志并调用通知（如 `notifyProjectMembers`），与 PRD“所有变动都要审批（审计）”“全员收到消息”一致。

### 4.7 自动备份（运维/脚本）

- [ ] **定时备份**：每 12 小时执行一次全量备份，保留 2 个月；使用 sqlite `.backup` 或等效安全方式，避免 WAL 未刷盘导致不完整。

## 五、交互与体验优化要点

- **实时感**：任何增删改后，列表/看板/时间轴/审批/通知 应在短时间内自动更新（SSE + invalidate），无需手动刷新。
- **加载与空态**：列表/详情加载中有骨架或 loading；无数据时用 EmptyState 等统一组件，文案清晰。
- **错误与反馈**：请求失败有 Toast 或明确提示；危险操作有二次确认（如删除任务/项目）。
- **导航一致**：同一入口（如从概览进入某项目）与从项目列表进入行为一致；返回后状态不错乱（如经理身份不丢失）。

## 六、审美与“非 AI 化”UI 原则

- **保留现有设计系统**：已采用 Plus Jakarta Sans / DM Sans、Slate 色板、阴影体系，避免替换为通用“AI 感”模板。
- **信息层级**：概览作为导航中枢，组件可点击穿透到详情，避免重复堆砌相同信息。
- **品牌化**：主色、圆角、间距、卡片样式保持统一，不引入与现有风格冲突的组件库或插图。
- **列表/卡片**：审批、通知等以列表或卡片承载清晰字段（项目名、操作人、任务、时间、改动），避免过于拥挤或过于空洞。
